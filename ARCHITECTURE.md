# Barbets — how it all works

  Barbets is a multi-tenant, play-money prediction market platform for private friend groups. Each group runs its own sealed token economy — no real money ever changes hands. Members create markets ("Will Jake finish the marathon?"), bet tokens on the outcome, and settle up via a parimutuel payout once the group resolves what actually happened. The standout feature is that a market can be *about* someone (`@mentioned` as a "subject"), and that person cannot see the market exists — not in their feed, not in counts, not in notifications — until it resolves.

Live at **https://mybarbets.com**. Installable as a PWA (Android + iOS) with real web push notifications.

## Tech stack

- **Next.js 16** (App Router, React 19, TypeScript) — server components for reads, Server Actions for all mutations
- **Supabase** — Postgres, Auth, Row Level Security, Storage (one private bucket), Edge Functions (Deno), pg_cron + pg_net for scheduled jobs
- **Tailwind CSS v4** — CSS-based `@theme`, no `tailwind.config.ts`
- **Vercel** — hosting, deploys via `vercel --prod`
- **Vitest** — integration tests that hit the real hosted Supabase project as real logged-in users (no mocks)

## Data model

Postgres enums define the closed vocabularies for the state machines:

- `market_type`: `yes_no` | `over_under` | `multiple_choice`
- `market_status`: `pending_sponsor` → `open` → `closed` → `proposed` → (`disputed`) → `resolved` | `voided`
- `bet_side`: `yes` | `no` | `over` | `under` (what you actually bet/vote/propose on for `yes_no`/`over_under` markets — `multiple_choice` uses an `option_id` instead, see below)
- `market_outcome`: `bet_side` + `void` (what a `yes_no`/`over_under` market resolves to, or a proposer can directly propose; a `multiple_choice` market resolves via `outcome_option_id` instead, leaving `outcome` null unless the result is VOID)
- `ledger_entry_type`: `seed` | `bet` | `payout` | `refund`
- `membership_status`: `active` | `dormant` | `removed`
- `season_status`: `active` → (`winding_down`) → `archived` → `intermission` → next season's `active`. `season_length`: `1m` | `2m` | `3m` | `custom` (a specific owner-picked end date/time) | `manual` (no fixed end)

Core tables: `users`, `groups`, `group_settings`, `memberships`, `seasons`, `season_optins`, `season_optouts`, `season_results`, `markets`, `market_subjects`, `market_options`, `bets`, `ledger`, `resolution_proposals`, `challenges`, `votes`, `market_reactions`, `resolution_clarifications`, `group_titles`, `push_subscriptions`, `notification_events`.

**Non-owner members get read-only visibility into `group_settings`** (betting on/off, token allocation, time zone, season status, whether hedged betting is allowed, the payout-distribution setting below) via the settings page — the owner is still the only one who can call `update_group_settings`; everyone else just gets to see what it's currently set to instead of it being owner-only screen real estate. Both `betting_enabled` and `seasons_enabled` are one-way switches — `update_group_settings` rejects flipping either back off once on, same reasoning both times: pausing things is what `end_season()` (voids+refunds open markets, opens intermission) is for, not a settings toggle. For a seasons-enabled group, `betting_enabled` stops mattering to `create_market()` entirely once the first season starts — see the per-season betting gate below. `allow_hedged_bets` (default on) is the one setting here that stays freely reversible in both directions — it only gates future `place_bet` calls, so there's no in-flight state either direction could corrupt, unlike the two one-way switches.

**Identity is per-group, not global.** `users` carries only `id` (auto-created on first sign-in, no separate claim step) and an unused `display_name`. What you're actually called and @mentioned as is `memberships.nickname` — one member can go by a different nickname in a different group. Set once on `create_group`/`join_group` (required on any path that inserts a brand-new membership row; reactivating a `dormant` row or no-op'ing on an `active` one keeps the existing nickname), changeable later via `update_nickname()`. Unique per group among non-`removed` memberships only, via a partial index (`memberships_group_nickname_unique`) rather than a plain constraint, so a `removed` member's old nickname frees up for someone else while a `dormant` member's stays reserved for them.

`group_settings.timezone` is an owner-set IANA zone name, purely informational: it's shown as a caption next to every "betting closes" field/display so members know what zone the person who set the closing time meant. It does not change how `datetime-local` inputs are interpreted (still the visiting browser's own local time) — there's no way to make that input read an arbitrary zone without real timezone-math, which would be a materially bigger feature.

**Multiple choice markets** add `market_options` (2–10 rows per market: `label` + `sort_order`, enforced in `create_market()`). Every place that used to carry a single `bet_side`/`market_outcome` value grew an `option_id` sibling instead — `bets.option_id`, `markets.outcome_option_id`, `resolution_proposals.proposed_option_id`, `votes.voted_option_id` — with exactly one of each pair ever populated on a given row, enforced by a same-table CHECK constraint (`..._xor_option`). VOID always goes through the original `outcome`/`bet_side` column as the literal `'void'`, never through `option_id` — VOID isn't an option, it's the automatic tie/zero-turnout fallback or an explicit proposal/ballot choice, same as for the other two types. Each option is a single field: plain text, or a leading `@nickname` that `create_market()` resolves against `memberships.nickname` in that group and attaches as the option's one subject (at most one subject per option, by construction — a label either starts with `@` or it doesn't). `market_subjects.option_id` is nullable and stays null for `yes_no`/`over_under` (market-level subjects, unchanged). Privacy stays market-level regardless of option — `is_market_visible()` matches on `market_id` alone, so being @'d in any single option still hides the whole market (seeing any option would reveal the market exists).

### The privacy choke point

Every table that can leak a subject's existence is gated by one function:

```sql
create function is_market_visible(p_market_id uuid, p_user_id uuid default auth.uid())
returns boolean ... security definer as $$
  select exists (
    select 1 from markets m
    join memberships mem on mem.group_id = m.group_id and mem.user_id = p_user_id
    where m.id = p_market_id
      and (
        not exists (select 1 from market_subjects ms where ms.market_id = m.id and ms.user_id = p_user_id)
        or m.status in ('resolved', 'voided')
      )
  );
$$;

create view visible_markets with (security_invoker = true) as
  select m.* from markets m where is_market_visible(m.id, auth.uid());
```

Every RLS policy on `markets`, `market_subjects`, `bets`, `resolution_proposals`, `challenges`, `votes`, and `market_reactions` calls this same function. `security_invoker = true` on the view is load-bearing — without it, the view would run with the *view owner's* privileges instead of the querying user's. RLS is deny-by-default: no table has a client-facing INSERT/UPDATE/DELETE policy at all. The only way to change money- or market-state is through the SECURITY DEFINER functions below.

A query filtered by RLS that comes back empty always means the same thing to the client — 404, never a distinguishable 403 — whether the row genuinely doesn't exist or you're a hidden subject. `lib/errors.ts`'s `notFoundIfEmpty()` enforces this at every route.

## Postgres functions (the only way to mutate anything)

All are `SECURITY DEFINER`, `SET search_path = public`, and explicitly `REVOKE`d from `public`/`anon` then `GRANT`ed only to `authenticated` (Supabase auto-grants EXECUTE by default, which would otherwise be an over-permissive hole).

- **`create_group` / `join_group` / `remove_member` / `leave_group`** — group membership. Leaving is self-service: it no longer refunds the leaver's own open bets (those stakes stay in their pools and settle normally into the now-dormant membership's balance), but still voids+refunds any market where the leaver is a **subject** (unavoidable — the market's premise left the room). Rejoining a `dormant` membership reactivates it in place with whatever balance is already sitting there — never reseeded; a `removed` membership (owner-initiated only) can never rejoin at all. `remove_member` still refunds the removed member's own open bets (they didn't choose to leave) and additionally rotates the group's invite code in the same transaction, so the removed member's known code is dead going forward. Seeding happens in exactly two places: first-time join, and season start for opted-in members.
- **`create_market` / `sponsor_market`** — a market needs two humans: whoever creates it, and a *different* member who endorses ("sponsors") it before it opens. Unsponsored markets auto-expire after 72h. Subject count is capped at `member_count - 2` (enough non-subject members left for a creator and a separate endorser — both of whom can also bet on it themselves). For `multiple_choice`, `create_market` instead takes an ordered list of 2–10 option strings (plain text, or `@nickname`), and both the creator/sponsor exclusion and the subject-count cap apply to the **union of distinct users resolved across every option** — a nickname can only be @mentioned in one option (checked case-insensitively, so `@dan` and `@Dan` collide even though they're distinct label text).
- **`place_bet`** — locks the market row (serializes against `expire_stale()` concurrently closing it) and the bettor's membership row. A bet can be for any amount from 1 up to the bettor's full current balance (no percentage cap). Dormant members and subjects can't bet. `multiple_choice` bets carry an `option_id` instead of a `side`; by default a member may hedge across multiple sides/options in the same market, but the owner can turn `group_settings.allow_hedged_bets` off, which rejects a bet on a side/option that's `is distinct from` every side/option the member already holds in that market — a same-side top-up (identical side/option to an existing bet, just more staked) always stays allowed either way, since only genuine hedging is the target.
- **`propose_resolution`** (8h challenge window starts) → **`challenge_resolution`** (moves to a secret ballot) → **`cast_vote`** → **`finalize_market`**.
  - `propose_resolution` may fire while a market is still `open`, not just `closed` — betting locks the instant the proposal commits (the same market-row lock `place_bet` already takes makes this race-safe). `closed_at` records when betting actually locked, independent of `closes_at` (which stays the *latest* possible close), so the UI can show "closed early by proposal." This early-close path emits only `resolution_proposed`, not also `market_closed` — recipients don't need "odds are live" and "a resolution was proposed" as two separate pushes for the same moment. The natural auto-close path (`expire_stale()` reaching `closes_at` with no proposal yet) is unaffected and still emits `market_closed` on its own, since nobody's proposed anything at that point.
  - A vote resolves either when the 8h vote window elapses, *or* the moment every eligible (non-subject, non-removed) member has voted — whichever comes first. `multiple_choice` ballots list every option plus VOID.
  - **Turnout rule**: zero ballots cast, or a tie for first place that *includes* the originally proposed outcome, upholds the proposal instead of voiding — apathy or indecision defaults to what was proposed, so challenging is only worth it if you can actually rally votes against it. A tie that *excludes* the proposal, or an explicit VOID majority, still resolves as **voided** (full refund to everyone) — the group actively disagreed with the proposal but couldn't agree on an alternative.
  - `propose_resolution` also takes an optional `p_photo_path`, a proof photo the proposer attaches at proposal time (`ProposeResolutionCard`, compressed client-side in `lib/compressImage.ts`, no dependency). It rides untouched through a challenge/vote (`resolution_proposals` is never re-inserted for a market, only its `finalized`/`votes_revealed_at` columns get updated) and stays visible after resolution — except on a `voided` market, since a void means the group never settled on what the photo was supposed to prove. The photo itself lives in the private `resolution-proofs` Storage bucket, which has zero `storage.objects` RLS policies (deny-by-default, same posture as every other table here) — every read and write goes through `lib/actions/resolution.ts`'s `proposeResolution()`/`getResolutionProofUrl()`, which re-check the exact same `is_market_visible`-backed RLS (via `resolution_proposals`/`visible_markets`, queried as the requesting user) before ever reaching for the service-role client to upload or sign a URL. One choke point, not two.
- **`request_clarification` / `update_resolution_criteria`** — lets a non-creator member flag that a market's resolution criteria (`markets.description`) is unclear while betting is still `open`, and lets the creator tighten the wording in response. Not secret while pending: `resolution_clarifications` is visible to everyone who can see the market (same `is_market_visible()`-gated, select-only RLS posture as `resolution_proposals`/`challenges`), so the question shows up on the market page like a proposal justification does. `update_resolution_criteria` is strictly reactive, never a free edit — it rejects a non-creator caller (`forbidden`, not subject-masked, since the creator can never be a subject of their own market by construction) and requires at least one existing row in `resolution_clarifications` for the market (every row in that table is currently-pending by construction — see below). A successful update **deletes every row** for that market in the same transaction, not just marks them answered — nothing lingers as history once addressed; the new `description` text is the only trace an answer ever happened. Both actions are gated to `status = 'open'` only, checked before any other business-rule check in `update_resolution_criteria`, so the gate holds even when a pending request still exists (e.g. an early resolution proposal locked betting mid-conversation). `clarification_requested` is single-recipient (the creator only, via a `get_event_recipients` carve-out like `member_joined`/`impressive_bet`); `criteria_updated` reaches everyone else who can currently see the market, same as `market_closed`. A market the current user created with an open row in `resolution_clarifications` also surfaces as a `needsClarificationResponse` bucket in `lib/waitingOnYou.ts` (a "Needs clarification" section on `/inbox`, counted in the header badge same as `needsEndorsement`/`awaitingVote`) and as a small "!" badge next to the status pill on the market card/detail page — both computed by an inner-join existence check against `resolution_clarifications`, since presence of any row already means pending. On the market page itself, asking is a circular "?" trigger in the resolution criteria card's corner that opens a small popover (explaining what asking does and what happens next) — the same overlapping-popover treatment `ReactionBar` uses on the reveal ticket, just re-themed for a light card instead of the dark ticket background.
- **Parimutuel payout** (`finalize_market`, the money-critical core): each winning bet's base payout is `floor(bet.amount * total_pool / winning_pool)`. The leftover "dust" from flooring goes entirely to the single largest winning stake (ties broken by earliest bet, then bet id). This guarantees `sum(payouts) == total_pool` exactly — no tokens created or destroyed. A one-sided market falls back to every bet winning outright (dust and all, same formula). The winner-filter (`side = v_winning_bet_side OR option_id = v_outcome_option_id`) is a single unified expression across all three market types — exactly one of those two variables is ever non-null for a given resolution, so `multiple_choice` reuses the identical payout/dust code path with no branching.
- **Winning side has zero bets ("everyone loses")** — nobody predicted the outcome that actually happened. By default this is a full refund of every stake, same as VOID. The owner can flip `group_settings.distribute_payout` on instead (default off; `creator_payout_pct`/`endorser_payout_pct`, default 25/5, must sum to ≤100): the market's creator and endorser take their cut of the real stakes on that market, and the remainder tops up `bonus_pool` equally across every other currently-*open* market in the group — a market absorbs its `bonus_pool` into its own `total_pool` the same way real stakes are absorbed, whenever it next resolves (win, its own zero-winner-pool split, or void). If there's no other open market to receive the remainder, it instead refunds proportionally back to this market's own bettors (same floor+dust construction as a normal winners split, just against the real pool instead of a winning side); if this market never had any real bettors either, it settles to the group owner. `refund_all_bets()` (used here when the setting is off, by every VOID resolution, and by season-end/member-removal force-voids) always drains a market's `bonus_pool` the same way on its way out — split across other open markets, or to the owner if none — so inherited bonus money can never get stranded on a market that never resolves to a real winner. Whenever this branch fires, `markets.payout_breakdown` (jsonb: `creator_cut`/`endorser_cut`/`other_markets_cut`/`refunded_to_bettors`/`settled_to_owner`, exactly one of the last three ever non-zero) records precisely where the pool went, for the reveal page — this can't be reconstructed from the ledger after the fact, since a `bonus_pool` transfer to another market is a pot-level column update, not a balance change, so it never gets a ledger row. Null for every other `finalize_market()` path.
- **`void_market_by_owner`** — an unconditional owner kill switch for one market, at any stage before it's already `resolved`/`voided` (`pending_sponsor` through `disputed`). Refunds every stake via the same `refund_all_bets()` helper VOID resolutions and season-end/member-removal force-voids already use (so it also correctly drains any inherited `bonus_pool`), then emits `market_voided` instead of `market_resolved` — a deliberately different push so recipients know the owner stepped in rather than assuming a proposal/vote ran its course. Not reachable if the owner happens to be the market's subject (`not_found`, same 404-not-403 rule as everywhere else).
- **`void_market_by_creator`** — the fallback for the one case `void_market_by_owner` can never reach: the owner is themselves `@mentioned` as the market's subject, so `is_market_visible()` hides the market from them before ownership is even checked. Gives the market's *creator* (who by `create_market`'s own constraint can never be a subject) an equivalent on-demand override, strictly gated to that one scenario — it 404s like every other subject-adjacent function, and refuses outright (`invalid_operation`, not a silent no-op) if the owner isn't actually a subject of this specific market, since the owner is still meant to be the one calling the shots whenever they're able to. Every other stuck-market path still resolves eventually on its own (a normal vote can VOID it, or `end_season()` force-voids everything), just not on demand.
- **`end_season` / `start_season` / `opt_in_season` / `opt_out_season` / `cancel_season_optout` / `open_season_betting` / `rename_season`** — seasons unify continuous play, long seasons, and short one-off event nights into one system: the only real differences are duration, and whether the group bothers with the ceremony below.
  - **Wind-down instead of instant force-void.** `end_season()` immediately force-voids any market that never got a resolution proposed (`pending_sponsor`/`open`/`closed` — nobody started the clock, nothing to wait on), but a market already `proposed`/`disputed` is left alone to run its normal 8h challenge/vote timers. If nothing was in flight, the season archives synchronously in the same call (the common case, and the only case before this). If something was, the season instead moves to `winding_down` (a `seasons.wind_down_deadline`, 8h past season end, is the hard cap) — `create_market()`'s `status = 'active'` lookup simply doesn't match `winding_down`, so no new markets can be created during it with no extra code. `_finalize_season()` (the shared internal helper doing the snapshot/archive/next-intermission-row work `end_season()` used to do inline) gets called from three places: `end_season()` itself when nothing's in flight, `finalize_market()`'s tail hook the instant it clears the *last* in-flight market of a `winding_down` season (so a season doesn't sit around for up to a minute waiting on the next cron tick), and `expire_stale()`'s hard-cap sweep, which force-voids anything still unresolved once `wind_down_deadline` passes and archives from there. `season_ended` is emitted with the real actor (the owner who clicked "End season now") only on the synchronous fast path; the deferred paths have no honest single actor and emit with `actor_id = null`.
  - **Opt-out by default, not opt-in.** A currently-**active** member is included in the next season automatically — `opt_out_season()`/`cancel_season_optout()` (writing to `season_optouts`, presence = "sitting this one out") is how they preemptively skip it, mirroring `opt_in_season`'s existing dual intermission/already-active behavior in the opposite direction. A currently-**dormant** member (self-service leave, a prior opt-out, or just joined mid-intermission) stays dormant unless they explicitly ask in — that's still exactly `opt_in_season()`/`season_optins` (presence = "asking to be swept in"), completely unchanged by this. The two tables are deliberately not one inverted table: `_cleanup_departing_member()` (shared by `leave_group`/`remove_member`) already deletes a departing member's `season_optins` row on every departure, and reusing that same delete for an "opted out" table would have silently *un-excluded* them instead — `start_season()`'s reseed rule is a union of `(active AND no season_optouts row)` OR `(dormant AND has a season_optins row)`, so a plain dormant member with neither row is correctly left alone either way.
  - **Betting starts paused every season**, including a group's very first one — `seasons.betting_open` (default `false`) is a separate gate from the group-level `betting_enabled` for exactly the groups where it matters (`seasons_enabled = true`); `open_season_betting()` (owner-only, once per season) is the explicit "go" that lets the owner see who's actually playing before any market can be created. Continuous (non-seasons) groups are untouched — `betting_enabled` alone still gates them.
  - **Duration**: `1m`/`2m`/`3m` (relative to `started_at`) or `custom` (`group_settings.season_custom_ends_at`, a specific date/time — useful for a single weekend or a summer with a real end in mind) get frozen onto `seasons.ends_at`/`season_length` the moment `start_season()` activates them (same freeze-at-start pattern `seed_amount` already used), which is also what `create_market()` checks `p_closes_at` against and what `expire_stale()`'s timed-end sweep reads — nothing here is recomputed live from `group_settings` mid-season, so editing settings mid-season can't retroactively move the current season's end. `manual` seasons have `ends_at is null`: no auto-end, no `closes_at` ceiling. A `custom` date that's since lapsed by the time `start_season()` actually runs (e.g. configured during a long intermission) is rejected with a clear "pick a new one" error rather than silently freezing a bogus already-past `ends_at`.
  - `rename_season()` (owner-only, any status) sets `seasons.name`; blank clears it back to the "Season N" display fallback. Purely cosmetic, no notification.
  - Seeding still happens in exactly the same two places as before: first-time join, and season start (now for the *default-included* population above, not a literal opt-in list).
- **`delete_group` / `cancel_group_deletion`** — deletion is grace-period, not immediate. `delete_group` (owner-only) voids+refunds every non-terminal market in the group right away (via the same `_void_market()` helper `end_season()` uses), schedules the actual row deletion 5 days out (`groups.deletion_scheduled_at`), and blocks new markets/members in the meantime (`create_market`/`join_group` both check the column) — so nothing new gets created only to be destroyed a few days later without anyone realizing why, and the group's final state stays visible/readable for the whole window. `cancel_group_deletion` clears the flag before the window elapses; it can't and doesn't un-void the markets already refunded, same as every other force-void in this app. `expire_stale()` sweeps up the actual `DELETE` once the grace period lapses (cascades through every group-scoped table). The same grace-period mechanism also fires automatically — `group_deletion_scheduled_inactivity` instead of `group_deletion_scheduled`, honest distinct copy since nobody actively did anything — when a group sits in `intermission` for 30 days with nobody continuing; `start_season()` clears it (and emits the ordinary `group_deletion_canceled`) the instant an owner does continue, since that really is what canceled it.
- **`react_to_market`** — one fixed emoji per person per market (`fire`/`laugh`/`clown`/`salute`/`thumbs_up`/`thumbs_down`), only ever on a `resolved`/`voided` market (gated by `is_market_visible()`, same choke point as everywhere else — a market's own subject can react too, once it resolves). Tapping the current pick again clears it; tapping a different one swaps it. No push notification. `lib/reactions.ts` holds the one canonical emoji/glyph mapping and display order, shared by the interactive picker (`ReactionBar`, on the reveal ticket) and a compact view-only facepile rendered on resolved/voided market rows/cards in the group hub — the latter shares its badge slot with the `needsAttention` "!" badge above, since a market can only ever be `open` (attention) or `resolved`/`voided` (reactions), never both.
- **Hall of Fame titles** (`group_titles`) — 8 persistent flavor badges (`lib/titles.ts`: The Oracle, Ice Cold, Bandwagon, Cursed, On Fire, Degenerate, Whale, Risk Taker) shown as flair next to a nickname (`<Mention titles={...}>`) everywhere, not just a leaderboard page. All 8 are computed from real (never `voided`) bet/ledger history, restricted to still-active memberships. 7 of them (`_recompute_group_titles`) are comparative rankings across the whole group and only recompute every 3rd market that resolves (`group_settings.markets_resolved_since_titles`, bumped by `_bump_titles_counter` from inside `finalize_market`) — recomputing on every single resolution would flip a title back and forth on a couple of noisy bets. Risk Taker (`_upsert_risk_taker`) is the exception: it's monotonic (only a genuine new all-time-best payout multiple ever changes the holder, the same stat the `impressive_bet` push already tracks), so it updates live on every resolution with no flapping to guard against, and replaced the old standalone "most impressive bet" leaderboard card. A vacant title (nobody yet qualifies, e.g. under the 5-settled-bet minimum) is still a row with `user_id = null`, so the UI can show "unclaimed" rather than not knowing whether it's ever been computed.
- **`expire_stale()`** — the cron entry point, scheduled via pg_cron every minute (`barbets-expire-stale`, same cadence as send-push below — tightened from an original 5-minute interval once send-push proved 1-minute pg_cron scheduling works fine on this project's plan). Covers 72h unsponsored expiry, market auto-close at `closes_at` (stamping `closed_at`), 8h no-challenge auto-finalize, 8h vote-window auto-finalize, the `winding_down` hard-cap sweep (force-void anything still `proposed`/`disputed` past `wind_down_deadline`, then archive), timed season auto-end (a plain `seasons.ends_at <= now()` check, not recomputed from `group_settings` — see above), the 30-day-abandoned-intermission sweep (schedules group deletion the same way `delete_group()` does), the grace-period group-deletion sweep, and a 30-day retention cleanup of already-`processed_at` rows in `notification_events` (that table otherwise has no retention policy at all — nothing else queries events by age, but there's no reason to keep them around forever). `get_cron_job_status()` (service_role only) reports both jobs' registered schedules for debugging, since `cron.job` isn't exposed via PostgREST directly.

## Money & the ledger

`ledger` is an append-only, signed log (`seed` / `bet` / `payout` / `refund`) — no client can ever UPDATE or DELETE it. Every balance change writes a matching ledger row in the same transaction that changes `memberships.balance`, so the ledger is always independently reconstructable and auditable.

## Server Actions & error handling

Every mutating action in `lib/actions/*.ts` calls a Postgres RPC and returns an `ActionResult<T>`:

```ts
type ActionResult<T> = { data: T; error?: undefined } | { data?: undefined; error: string };
```

**This is load-bearing, not a style choice.** Next.js redacts any error *thrown* out of a Server Action in production — the client only ever sees a generic message + a digest, never the real text, even for an expected rejection like "betting is not open on this market." Actions must catch the Postgres error and *return* it instead; client components check `result.error` rather than wrapping the call in try/catch. `lib/errors.ts`'s `runRpc()` does this conversion and also strips the internal `invalid_operation:` / `not_found:` style prefix so the user sees plain text, not what looks like an error code.

Read-only Server Components query `visible_markets` and read-only RPCs directly — those still throw/404 via `notFoundIfEmpty()`, since that path isn't subject to the same Server Action redaction.

## Notifications

`notification_events` is an internal queue table (no RLS policies at all — only `service_role` touches it) written transactionally by whichever Postgres function causes a notifiable transition. This is deliberate: several transitions (market auto-close, auto-finalize, auto-end-season) only ever happen via the `expire_stale()` cron job, so hooking notifications into Next.js Server Actions would silently miss most of them. Writing the event *inside* the same Postgres transaction that changed the state means "this happened, exactly once" is transactionally safe, decoupled from "send a push" (network I/O, retryable).

`get_event_recipients(event_id)` is the single choke point for who gets notified: it reuses the same subject-exclusion logic as `is_market_visible`, filters to members with a live `push_subscriptions` row and `notifications_enabled = true`, and excludes the triggering actor (`actor_id`) if there is one. Notably, `finalize_market()`'s actor is just `auth.uid()` — when a voter's ballot auto-finalizes the market in the same request, that's still their session, so they're correctly excluded from the "market resolved" push about their own action; cron-triggered finalizes have no request context, so `auth.uid()` is null and nobody is excluded on that basis. `betting_opened` and `season_ended` are group-scoped (every non-removed member, actor excluded) rather than market-scoped. `impressive_bet` repurposes `actor_id` to mean the opposite of everywhere else: the one person to notify, not the one to exclude — `finalize_market()` sets it to the bettor whose payout just became the group's new all-time-best payout multiple (`payout ÷ amount`, see `get_most_impressive_bet()`), and only they get the push. `member_joined` is also group-scoped but single-recipient like `impressive_bet` — its one recipient is always `groups.owner_id`, not the actor, so a group with a large roster doesn't get a push blast every time someone joins; only `join_group()`'s genuinely-new-membership paths emit it, not a dormant member's rejoin. `market_voided` is market-scoped like `market_resolved` and joins it in the subject-inclusion branch (`p_include_subjects`) for the same reason: voiding lifts `is_market_visible()`'s privacy gate just like resolving does, so subjects are very much part of the intended audience, not excluded from it.

A scheduled Supabase Edge Function (`supabase/functions/send-push`, Deno + `web-push`, run every minute via pg_cron + pg_net) drains up to 50 unprocessed events per run, computes recipients, and sends real Web Push notifications. Dead subscriptions (404/410 from the push service) are cleaned up automatically.

## PWA & push

- `public/manifest.json`: `start_url`/`scope` are both `"/"` (not a page that requires auth — Android's install criteria need a reachable start URL), standard + maskable icons.
- Maskable icons must be a full-bleed square with a *solid* background (no transparency, no pre-baked rounded corners) — Android applies its own mask shape at display time, and content needs to sit within the inner ~80% "safe zone." A rounded, padded source image shows up with blank corners once Android's own mask is applied over it.
- The Android push **badge** (small status-bar icon) is a *separate*, dedicated asset (`badge-mono.png`) — Android renders it as a solid-tint silhouette using only the alpha channel, ignoring color entirely. The regular full-color app icon is used for the larger in-notification `icon` field, which platforms that support it still show in color.
- `public/sw.js`: network-first for everything (this app is almost entirely dynamic — a stale cached market page could show wrong odds), with a resilient `Promise.allSettled` app-shell precache so one failed asset can't block install. Bump `CACHE_NAME` whenever `SHELL_URLS` changes.
- iOS never fires `beforeinstallprompt`. `components/pwa/PushSetup.tsx` detects iOS + not-yet-installed and shows a Share → Add to Home Screen walkthrough instead of a broken permission prompt; this triggers automatically, nothing to configure.
- There's no app-wide interrupt nudging people to subscribe (the old `PushReminderModal` was removed) — `PushSetup` lives permanently on `/profile` instead, the one place to opt in or out.
- VAPID private key lives only as a Supabase Edge Function secret, never in Next.js env or the client bundle.

## Notable design decisions worth remembering

- **404-not-403 everywhere.** A hidden subject and a genuinely nonexistent market must be indistinguishable, at every layer, forever.
- **Business-rule errors return, they don't throw**, specifically inside Server Actions — see the Server Actions section above. This bit us once already (a bet-timing race showed as a raw "server error" instead of friendly copy) before the whole action layer was converted.
- **Ledger is append-only** with no client mutation policy, as defense in depth on top of RLS already blocking it.
- **Subject cap is `member_count - 2`**, not `- 3` — the creator and the endorser are the only two roles that strictly can't overlap with the subject set; both can also bet on their own market, so no third "bettor" role needs reserving. For `multiple_choice`, this cap applies to the union of distinct subjects across every option, not per option.
- **The vote window and challenge window are both 8h**, and a vote auto-finalizes early the moment everyone eligible has voted rather than always waiting out the clock.
- **A resolution proposal can beat the clock.** `propose_resolution` accepts a market that's still `open`, not just `closed` — the instant the proposal commits, betting locks for everyone. `closes_at` stays the *latest* possible close; `closed_at` records when it actually happened, so the UI can distinguish "closed early by proposal" from a normal timeout. There's deliberately no reopen mechanism — a premature proposal that turns out to be wrong just gets challenged and voted VOID like any other bad proposal.
- **Leaving a group no longer refunds your open bets** — only being a market's *subject* forces a void+refund on leave/removal. This closes an exploit where a member could peek at closed-market odds, then leave to claw back a stake they knew was going to lose. A self-service leave goes to `dormant` (rejoinable, balance preserved, never reseeded); an owner-initiated removal goes to `removed` (permanent) and rotates the invite code.
- **Turnout apathy upholds the proposal, it doesn't void.** Before this, a tie or zero-turnout vote always voided the market — which made challenging a free "undo" for a losing bettor betting on the group not bothering to vote. Now the proposal wins a tie it's part of, and wins outright if nobody votes; a tie that excludes the proposal (the group disagreed but couldn't agree on what's true instead) still voids.
- **Multiple choice markets generalize via `option_id`, not a new outcome enum.** Every column that used to hold a `bet_side`/`market_outcome` value grew an `option_id` sibling with a same-table XOR CHECK constraint, so `finalize_market`'s money-critical payout math is one shared code path across all three market types instead of a per-type branch — the one place a bug would be catastrophic stays the one place the logic lives.
- **`ledger.bet_id` is optional for `reason = 'payout'`, unlike `bet`/`refund` which always require it.** Added for the `distribute_payout` creator/endorser cuts and the orphaned-`bonus_pool` owner fallback — both are market-level rewards with no single bet behind them. `market_id` is still required for every non-`seed` row.
- **`bonus_pool` money is redistributed, never re-taxed.** When a zero-winner-pool market's remainder tops up another market's `bonus_pool`, that inherited money rides through untouched by that second market's own `distribute_payout` cut if it also happens to hit its own zero-winner-pool case — only real stakes are ever taxed. This keeps the payout math from decaying every time money hops between markets.
- **Nicknames are per-group, not global.** The old design claimed a single global `username` at signup, gating every other action behind that claim. Real usage showed people wanted to go by different names in different friend groups, so identity moved onto `memberships.nickname` (set on join/create, changeable via `update_nickname`) and the claim step disappeared entirely — a profile row is just auto-created on first sign-in.
- **A group's time zone is a display caption, not an input-interpretation feature.** It tells members what zone the person who set a closing time meant; it does not make `datetime-local` inputs read differently per visitor. Revisit this if real usage shows people actually need the input itself to be zone-aware.
- **Deleting a group is grace-period, not instant.** `delete_group` voids+refunds every open market immediately (so nobody's bet/vote is left in limbo) but only schedules the row deletion 5 days out, blocking new markets/members in the meantime and letting the owner `cancel_group_deletion` before it lands. Nothing in `place_bet`/`propose_resolution`/`cast_vote`/etc. needed to change for this to be safe — every market in a group pending deletion is already `voided`, so anything already in flight just hits the same existing "market isn't open" rejection those functions always had.
- **Hall of Fame titles use two different recompute cadences on purpose.** 7 of the 8 are comparative rankings that only recompute every 3rd market resolution, so a title doesn't flap back and forth on a couple of noisy bets; Risk Taker updates live because it's monotonic (a new holder only ever appears via a genuine new all-time-best payout multiple), so there's nothing to flap in the first place. Don't reflexively make a new title recompute live "for freshness" — check whether it's actually monotonic first.
- **The reveal page is a shareable image, not just a page.** `RevealTicket.tsx` captures itself to a PNG via `html-to-image` the moment it mounts (not lazily on click) — waiting until a share tap would put an `await` between the user's click and `navigator.share()`, and Safari revokes the share sheet's required user-activation across an awaited gap. Any image inside that captured node (the coin logo) has to be a plain `<img>`, never `next/image`, since `html-to-image` needs a same-origin asset with no optimization endpoint in the way.
- **Resolution criteria can only change in response to an open clarification request, never as a free edit, and only while the market is still `open`.** This closes the obvious exploit (a creator quietly rewording a wobbly market after odds are visible) without blocking legitimate tightening — the creator can always act once *someone* has flagged the ambiguity, and one update satisfies every pending flag at once rather than requiring them to be dismissed one by one.
- **Storage is the one place access control isn't a Postgres RLS policy.** The `resolution-proofs` bucket (resolution proposal proof photos) is the app's only use of Supabase Storage, and deliberately carries no `storage.objects` policies at all — `storage.objects` isn't gated the same way a table row is, so a photo's bytes don't automatically inherit `resolution_proposals`' RLS the way a new column on that table would. Rather than write a second, parallel access-control system in storage policies, access is mediated entirely by Server Actions in `lib/actions/resolution.ts` that query `resolution_proposals`/`visible_markets` as the requesting user (real RLS, real gate) before ever using the service-role client to upload or mint a signed URL. If a second Storage bucket is ever added, default to this pattern over writing bucket-level RLS.
- **Adding a new trailing default parameter to a Postgres function does not replace an existing overload — and this has bitten this project four separate times, most recently `update_group_settings` picking up a fourth new parameter across three different features.** A function's identity in Postgres is its argument *type* signature, not its defaults — `CREATE OR REPLACE` with a longer signature creates a second overload alongside the old one, and PostgREST then can't pick a candidate for a call that omits the new parameter (and worse, the error only surfaces the next time someone calls it *without* the new param — a function only ever touched with the new signature looks completely fine until then). Every signature change needs an explicit `DROP FUNCTION <old signature>` for the superseded version in the *same migration* that adds the parameter, not a "add first, drop unlisted the moment it breaks" approach. After pushing, verify with a throwaway call passing every parameter by name (`supabase.rpc(name, {...all params...})`) — if PostgREST error-message-dumps two candidate signatures, an overload was left behind.
- **Hedging is opt-out per group, not a global rule.** `group_settings.allow_hedged_bets` defaults to on (today's parimutuel behavior, unchanged for every existing group) specifically to close a real exploit: betting big on a heavy favorite and a token amount on the underdog manufactures a favorable risk profile regardless of outcome. The enforcement lives in `place_bet` itself; `BetslipBar`'s client-side block (disabling the confirm button when a selection conflicts with an existing bet) is just a courtesy, same relationship every other business rule here has to its UI.
- **`BetslipBar` replaced the old inline `PlaceBetCard`** with a persistent bottom bar, styled like the reveal ticket, that slides up into a confirmation sheet instead of a form sitting mid-page. The point is a bettor always sees a clear stub (side, amount, balance after) before confirming, not a form that just resets silently on submit — a successful bet hands off to a full-screen success view (small checkmark animation, respects `prefers-reduced-motion`) before returning to the refreshed market page. Quick-amount chips are 1%/5%/10% of `group_settings.seed_amount` (the tokens a new member starts with), rounded to the nearest 5, not a percentage of current balance — balance swings with every bet, seed amount doesn't, so it's the stable reference point for "a small/medium/large bet in this group."
- **Tailwind v4's `space-y-*` puts margin on the bottom of every child except the literal last one, not margin-top on all-but-first like v3.** A `position: fixed` element still receives that margin even though it's out of normal flow, and for a `bottom-0` element an unwanted margin-bottom shows up as a real gap between it and the true screen edge. This bit `BetslipBar` for real: it renders a Fragment of four top-level elements (an in-flow spacer, the closed bar, a backdrop, and the sheet) into a `space-y-*` container, and the *sheet* ends up as the container's actual last child, not the bar — so only the bar got the stray margin, which is exactly why it floated while the sheet always sat flush. Confirmed by reproducing both states (24px gap without a fix, 0px with) before shipping the fix: every top-level element `BetslipBar` renders now carries an explicit `!m-0`, rather than relying on which one happens to land last in the DOM.
- **The market page's "Closes in" stat tile signals urgency with a breathing danger-colored border, not honey.** Honey is already this app's one accent color (the "Line" tile for over/under markets, every primary button), so reusing it to also mean "hurry up" would blur two different meanings into one color — `danger` (already used for form errors) reads as "pay attention" without colliding with it. `ClosesInStatTile.tsx` computes the under-1h threshold client-side on the same 30s tick `CountdownTimer` already uses, rather than a value frozen at page load, so it turns on live for anyone watching a market close in real time instead of requiring a refresh. The border (`.animate-urgent-breathe` in `globals.css`) fades to a static `danger-500` outline under `prefers-reduced-motion`, same posture as the bet-confirmed checkmark and `RevealTicket`'s capture guard.
- **A season ending doesn't mean every in-flight market gets discarded, but it also doesn't wait forever.** The old `end_season()` force-voided anything non-terminal the instant it ran, which meant a market someone had already started resolving (proposed, maybe already challenged) got thrown away along with everything else. `winding_down` splits the two cases: nothing-started-yet force-voids immediately (there's nothing to wait on), already-in-flight gets to finish through its normal 8h timers, capped 8h past season end so a stuck vote can't hold a season open indefinitely. The three call sites that can trigger the actual archive (`end_season`'s synchronous fast path, `finalize_market`'s tail hook, `expire_stale`'s hard-cap sweep) all funnel through one shared `_finalize_season()` helper, same "one place the logic lives" instinct as the multiple-choice payout unification above.
- **`season_optins` and `season_optouts` are two separate tables on purpose, not one table with inverted meaning.** The obvious design once seasons flipped to opt-out-by-default would be to rename `season_optins` and read "no row" as "included." That breaks `_cleanup_departing_member()` (shared by `leave_group`/`remove_member`), which already deletes a departing member's opt row on every departure — under an inverted single table, that delete would silently *un-exclude* them instead of excluding them, auto-reseeding someone who explicitly left. Keeping both tables (`season_optins`: a dormant member asking in; `season_optouts`: an active member asking out) means the existing departure cleanup, `join_group`, and `opt_in_season` needed zero changes — `start_season()`'s reseed rule is just a union of the two default populations.
